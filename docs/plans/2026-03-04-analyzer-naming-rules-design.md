# SQL 分析器命名规范规则设计

**日期**: 2026-03-04
**作者**: Claude
**状态**: 已批准

## 概述

为 Spark SQL 静态分析器添加三条代码规范类规则，用于检测表名日期、注释格式和敏感信息。

## 新增规则

### 1. TABLE_NAME_WITH_DATE (表名日期检测)

**目的**: 检测表名中包含硬编码日期的情况

**严重级别**: HIGH

**检测模式**: 表名中包含 8 位日期格式（如 `tab_20250102_xxx`）

**正则表达式**:
```regex
\b(?:FROM|JOIN|TABLE|INSERT\s+INTO|UPDATE|CREATE\s+TABLE)\s+(?:\w+\.)?(\w*[12]\d{3}(0[1-9]|1[0-2])(0[1-9]|[12]\d|3[01])\w*|(?:\w+_)*[12]\d{3}(0[1-9]|1[0-2])(0[1-9]|[12]\d|3[01])(?:_\w+)*)
```

**提示信息**: "检测到表名中包含日期（如 table_20250102），请检查是否为分区表命名或临时表"

**建议**: "建议使用分区表规范命名，如 table_name PARTITION(dt='20250102')，或使用变量替代硬编码日期"

---

### 2. COMMENT_WITH_SEMICOLON (注释分号检测)

**目的**: 检测注释行中包含分号的情况，可能导致解析错误

**严重级别**: HIGH

**检测模式**: 行注释 `--` 后面包含 `;` 字符

**正则表达式**:
```regex
--[^;\n]*;
```

**提示信息**: "检测到注释行中包含分号（;），可能导致SQL解析错误"

**建议**: "请检查注释内容，如果分号是注释说明的一部分，请确保不会影响SQL解析；如果是误写的分号，请删除"

---

### 3. SENSITIVE_INFO_DETECTED (敏感信息检测)

**目的**: 检测 SQL 中的敏感信息（身份证号和相关字段名）

**严重级别**: HIGH

**检测模式**:
- 18位身份证号：`110101199001011234`
- 15位身份证号：`110101900101123`
- 敏感字段名：`id_card`, `idcard`, `identity`, `sfz`, `身份证`, `证件号`

**正则表达式**:
```regex
\b(?:1[1-9]\d{5}(18|19|20)\d{2}(0[1-9]|1[0-2])(0[1-9]|[12]\d|3[01])\d{3}[\dXx]|[1-9]\d{5}\d{2}(0[1-9]|1[0-2])(0[1-9]|[12]\d|3[01])\d{3})\b|(?:身份证|证件号|id_card|idcard|identity|sfz)
```

**提示信息**: "检测到可能的敏感信息（身份证号或相关字段名），请确保符合数据安全规范"

**建议**: "建议对敏感字段进行脱敏处理（如 md5/hash），或确保已获得相应授权；检查字段命名是否符合安全规范"

---

## 实现方式

采用**纯正则表达式**实现，与现有 16 条规则风格一致，保持代码简洁性。

规则添加位置：`backend/core/analyzer.py` 的 `_init_rules()` 方法

## 测试用例

| 测试场景 | 输入 | 预期结果 |
|---------|------|---------|
| 表名日期-后缀 | `FROM tab_name_20250102` | 触发 TABLE_NAME_WITH_DATE |
| 表名日期-中间 | `FROM tab_20250102_xxx` | 触发 TABLE_NAME_WITH_DATE |
| 注释分号 | `-- 这是注释;有问题` | 触发 COMMENT_WITH_SEMICOLON |
| 身份证号(18位) | `id_card = '110101199001011234'` | 触发 SENSITIVE_INFO_DETECTED |
| 身份证号(15位) | `id_card = '110101900101123'` | 触发 SENSITIVE_INFO_DETECTED |
| 敏感字段名 | `SELECT idcard FROM table` | 触发 SENSITIVE_INFO_DETECTED |
| 正常SQL | `FROM normal_table` | 不触发任何规则 |
